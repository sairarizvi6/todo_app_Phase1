## Architecture Design

### Layer Separation

#### 1. Models Layer (`models.py`)
**Purpose**: Data structure definitions
**Responsibility**: Define Task dataclass with validation
**No**: Business logic, storage operations, UI code

#### 2. Storage Layer (`storage.py`)
**Purpose**: Data persistence (in-memory)
**Responsibility**: CRUD operations on task dictionary
**No**: Business logic, validation, UI code

#### 3. Service Layer (`service.py`)
**Purpose**: Business logic
**Responsibility**: Task operations, validation, error handling
**No**: Direct storage access, UI code

#### 4. CLI Layer (`cli.py`)
**Purpose**: User interface
**Responsibility**: Display menus, get input, show results
**No**: Business logic, direct storage access

#### 5. Main (`main.py`)
**Purpose**: Application entry point
**Responsibility**: Initialize components, start CLI loop
**No**: Business logic, storage, UI rendering

### Data Flow
```
User Input ‚Üí CLI ‚Üí Service ‚Üí Storage ‚Üí Task Data
            ‚Üì       ‚Üì         ‚Üì
         Display ‚Üê Result ‚Üê Success/Error
```

## Feature Requirements

### 1. Add Task
- **Input**: Title (required), Description (optional)
- **Validation**:
  - Title: 1-200 characters, cannot be empty
  - Description: max 1000 characters
- **Output**: Confirmation with auto-generated Task ID
- **Error Handling**: Empty title, invalid characters

### 2. View Task List
- **Display**: ID, Title, Status (‚úì/‚òê), Created Date
- **Format**: Clean table or structured list
- **Empty State**: Friendly message when no tasks exist
- **Summary**: Show count (total, pending, completed)

### 3. Update Task
- **Input**: Task ID + (new title OR new description)
- **Validation**: Task ID must exist
- **Output**: Confirmation of updated fields
- **Error Handling**: Invalid ID, empty input

### 4. Delete Task
- **Input**: Task ID
- **Validation**: Task ID must exist
- **Confirmation**: Ask "Are you sure? (y/n)"
- **Output**: Confirmation message with deleted task title
- **Error Handling**: Invalid ID, invalid confirmation

### 5. Mark as Complete
- **Input**: Task ID
- **Toggle**: Pending ‚Üí Complete OR Complete ‚Üí Pending
- **Output**: Confirmation with new status
- **Error Handling**: Invalid ID

## Error Handling Strategy

### Input Validation
1. **Menu Choice**: Must be 1-6, handle non-numeric input
2. **Task ID**: Must be integer, must exist in storage
3. **Title**: Must not be empty, max 200 chars
4. **Description**: Max 1000 chars (optional)

### Error Message Format
```python
# Good error messages
"Invalid option. Please enter a number between 1-6"
"Task not found with ID: 5. Use option 2 to view all tasks"
"Title is required and must be 1-200 characters"

# Bad error messages (avoid)
"Error!"
"Invalid input"
"Something went wrong"
```

## Data Model Specification

### Task Structure
```python
 @dataclass
class Task:
    id: int                    # Auto-incrementing, starts from 1
    title: str                 # Required, 1-200 characters
    description: str           # Optional, max 1000 characters
    completed: bool            # Default: False
    created_at: datetime       # Auto-generated on creation
    updated_at: datetime       # Auto-updated on modification
```

### Storage Structure
```python
# In-memory storage
tasks: Dict[int, Task] = {}
next_id: int = 1
```
